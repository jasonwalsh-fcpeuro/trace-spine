* Overview

The Feature Flags Service provides flag evaluation, targeting rules, and percentage
rollouts for the Petstore Commerce Platform. All evaluations carry traceparent context
for A/B correlation — enabling causally-linked experiment analysis.

** Key Design Principles

1. *Trace-aware evaluation* — every flag decision includes trace-id for downstream
   correlation with metrics, conversions, and user behavior
2. *Redis-cached* — hot path reads from Redis; writes sync to PostgreSQL
3. *Deterministic rollouts* — percentage rollouts use consistent hashing on user-id +
   flag-key for stable assignment
4. *LaunchDarkly optional* — can sync flags bidirectionally or run standalone

** Components (from C4)

| Component       | Responsibility                          |
|-----------------+-----------------------------------------|
| Flag Evaluator  | Evaluates flags for given context       |
| Targeting Engine| User/segment targeting rules            |
| Rollout Manager | Percentage rollouts, schedules          |
| External Sync   | Syncs with LaunchDarkly if configured   |
| Admin API       | Flag CRUD operations                    |

** Targeting Operators

| Operator  | Description                  | Example                                   |
|-----------+------------------------------+-------------------------------------------|
| eq        | Equals                       | {"attribute": "user.country", "op": "eq", "values": ["US"]} |
| neq       | Not equals                   | {"attribute": "user.tier", "op": "neq", "values": ["free"]} |
| in        | In list                      | {"attribute": "user.tier", "op": "in", "values": ["gold", "plat"]} |
| not-in    | Not in list                  | {"attribute": "user.status", "op": "not-in", "values": ["banned"]} |
| contains  | String contains              | {"attribute": "user.email", "op": "contains", "values": ["@acme.com"]} |
| starts-with| String starts with          | {"attribute": "user.id", "op": "starts-with", "values": ["test_"]} |
| ends-with | String ends with             | {"attribute": "user.email", "op": "ends-with", "values": [".edu"]} |

| gte       | Greater than or equal        | {"attribute": "user.orders", "op": "gte", "values": [10]} |
| lt        | Less than                    | {"attribute": "cart.total", "op": "lt", "values": [100]} |

* Trace-Aware A/B Correlation

** How Traces Enable Experiment Analysis

The Feature Flags Service embeds trace-id in every evaluation result. This enables:

1. *Impression tracking* — Metrics Platform records which flag value was served,
   keyed by trace-id
2. *Conversion correlation* — when an order completes, its trace-id links back to
   the flag evaluation that determined the checkout variant
3. *Funnel analysis* — follow trace-id from flag evaluation through cart, checkout,
   payment to see how each variant affects completion rate

** Experiment Assignment Stability

To ensure users see consistent flag values across requests:

1. Rollout uses consistent hashing: ~murmurhash(bucket-by + flag-key + seed)~
2. Hash is computed once per user+flag, result is stable
3. Changing the seed rebalances all users (use for new experiment cohorts)

* Database Schema

** PostgreSQL Tables

#+begin_src sql :eval no
-- Flags table
CREATE TABLE flags (
    key             VARCHAR(255) PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    description     TEXT,
    kind            VARCHAR(20) NOT NULL DEFAULT 'boolean',
    default_value   JSONB NOT NULL,
    enabled         BOOLEAN NOT NULL DEFAULT false,
    archived        BOOLEAN NOT NULL DEFAULT false,
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Targeting rules
CREATE TABLE flag_rules (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    flag_key        VARCHAR(255) NOT NULL REFERENCES flags(key),
    description     TEXT,
    clauses         JSONB NOT NULL,          -- [{attribute, op, values}...]
    match_type      VARCHAR(10) NOT NULL DEFAULT 'all', -- all | any
    variation       JSONB NOT NULL,          -- the value when rule matches
    weight          INTEGER,                 -- null = 100%, or 0-100
    position        INTEGER NOT NULL,        -- evaluation order
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_flag_rules_flag_key ON flag_rules(flag_key);
CREATE INDEX idx_flag_rules_position ON flag_rules(flag_key, position);

-- Rollout configuration
CREATE TABLE flag_rollouts (
    flag_key        VARCHAR(255) PRIMARY KEY REFERENCES flags(key),
    enabled         BOOLEAN NOT NULL DEFAULT false,
    percentage      INTEGER NOT NULL DEFAULT 0,
    bucket_by       VARCHAR(255) NOT NULL DEFAULT 'user.id',
    seed            VARCHAR(255),
    variations      JSONB,                   -- [{value, weight}...]
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Rollout ramps (gradual increase)
CREATE TABLE flag_ramps (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    flag_key        VARCHAR(255) NOT NULL REFERENCES flags(key),
    current_pct     INTEGER NOT NULL,
    target_pct      INTEGER NOT NULL,
    step_pct        INTEGER NOT NULL,
    step_interval   INTERVAL NOT NULL,
    pause_on_error  FLOAT,
    status          VARCHAR(20) NOT NULL DEFAULT 'active', -- active | paused | completed | cancelled
    next_step_at    TIMESTAMP WITH TIME ZONE,
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_flag_ramps_next_step ON flag_ramps(status, next_step_at);

-- Evaluation audit log (optional, for debugging)
CREATE TABLE flag_evaluations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    flag_key        VARCHAR(255) NOT NULL,
    trace_id        VARCHAR(32) NOT NULL,
    user_id         VARCHAR(255),
    context_hash    VARCHAR(64),
    value           JSONB NOT NULL,
    reason          VARCHAR(20) NOT NULL,
    rule_id         UUID,
    evaluation_ms   FLOAT,
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Partition by time for efficient cleanup
CREATE INDEX idx_flag_evaluations_trace ON flag_evaluations(trace_id);
CREATE INDEX idx_flag_evaluations_created ON flag_evaluations(created_at);

-- LaunchDarkly sync state
CREATE TABLE launchdarkly_sync (
    flag_key        VARCHAR(255) PRIMARY KEY REFERENCES flags(key),
    ld_key          VARCHAR(255) NOT NULL,
    ld_version      INTEGER,
    last_synced_at  TIMESTAMP WITH TIME ZONE,
    sync_direction  VARCHAR(20) NOT NULL DEFAULT 'pull', -- pull | push | bidirectional
    conflict_at     TIMESTAMP WITH TIME ZONE,
    conflict_detail JSONB
);
#+end_src

** Redis Cache Structure

#+begin_src text
# Flag definitions (hash)
flags:checkout-v2 -> {
  "key": "checkout-v2",
  "kind": "boolean",
  "default": false,
  "enabled": true,
  "rules": [...],
  "rollout": {...},
  "version": 42
}

# Flag version (for cache invalidation)
flags:checkout-v2:version -> 42

# All flag keys (set, for batch operations)
flags:keys -> {"checkout-v2", "express-pay", "loyalty-widget", ...}

# Evaluation cache (optional, with TTL)
eval:checkout-v2:hash(context) -> {value, reason, expires_at}
TTL: 60s

# Ramp state
ramp:checkout-v2 -> {current_pct, target_pct, next_step_at}
#+end_src
