* Overview

The Payment Service orchestrates payment processing for the Petstore Commerce Platform.
It coordinates between payment processors (Stripe, PayPal), the Wallet Service (store credit),
and the Fraud Service (pre-authorization risk scoring).

** Core Responsibilities

1. Payment orchestration across multiple processors
2. Idempotency key handling to prevent duplicate charges
3. Retry with exponential backoff for transient failures
4. Circuit breaker patterns for processor outages
5. Transactional outbox for reliable event publishing
6. Trace context propagation through all payment flows

** Trace Propagation Contract

All payment flows MUST preserve trace-id across:
- Incoming HTTP requests from Order Service
- Outbound HTTP calls to Stripe/PayPal
- Outbound HTTP calls to Wallet Service
- Outbound HTTP calls to Fraud Service
- Outbox events published to Kafka

Every span created by this service MUST be a child of the incoming trace context.

* Payment Orchestration Flows

** Primary Payment Flow

#+begin_src mermaid :file payment-flow.png :eval never-export
sequenceDiagram
    participant Order as Order Service
    participant Pay as Payment Orchestrator
    participant Idem as Idempotency Layer
    participant Fraud as Fraud Service
    participant Wallet as Wallet Service
    participant Stripe as Stripe Adapter
    participant Outbox as Outbox Publisher

    Note over Order,Outbox: traceparent: 00-{trace_id}-{parent_span}-01

    Order->>+Pay: POST /charge (idempotency_key, amount, wallet_credit)
    Pay->>Pay: extract_trace_context(headers)
    Pay->>Pay: create_child_span("payment.orchestrate")

    Pay->>+Idem: check_idempotency_key(key)
    Idem-->>-Pay: {status: new|duplicate|in_progress}

    alt duplicate
        Pay-->>Order: {status: success, charge_id: cached}
    else in_progress
        Pay-->>Order: 409 Conflict (retry with backoff)
    else new
        Idem->>Idem: mark_in_progress(key)

        Pay->>+Fraud: POST /score (inject traceparent)
        Fraud-->>-Pay: {risk_score: 0.15, decision: allow}

        alt risk_score > threshold
            Pay->>Idem: mark_failed(key, "fraud_rejected")
            Pay-->>Order: {status: declined, reason: fraud}
        else risk acceptable
            alt wallet_credit > 0
                Pay->>+Wallet: POST /apply (inject traceparent)
                Wallet-->>-Pay: {applied: $50, remaining: $150}
            end

            Pay->>+Stripe: POST /charges (inject traceparent)
            Stripe-->>-Pay: {charge_id: ch_xxx, status: success}

            Pay->>+Outbox: record_event(payment.completed, traceparent)
            Outbox-->>-Pay: {event_id: uuid}

            Pay->>Idem: mark_complete(key, result)
            Pay-->>-Order: {status: success, charge_id: ch_xxx}
        end
    end
#+end_src

* Idempotency Key Handling

** Key Format

#+begin_src text
{client_id}-{order_id}-{attempt_number}
#+end_src

Example: ~storefront-ord_12345-1~

** State Machine

stateDiagram-v2
    [*] --> new: key_not_found
    new --> in_progress: lock_acquired
    in_progress --> complete: payment_success
    in_progress --> failed: payment_failed
    complete --> [*]
    failed --> [*]

#+end_src

** Database Schema

#+begin_src sql
CREATE TABLE idempotency_keys (
    key             VARCHAR(255) PRIMARY KEY,
    status          VARCHAR(20) NOT NULL DEFAULT 'in_progress',
    request_hash    VARCHAR(64) NOT NULL,
    response_body   JSONB,
    traceparent     VARCHAR(55) NOT NULL,
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    expires_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW() + INTERVAL '24 hours'
);

CREATE INDEX idx_idempotency_expires ON idempotency_keys(expires_at)
    WHERE status = 'in_progress';
#+end_src

** Conflict Resolution

| Scenario | Action |
|----------|--------|
| Key exists, status=complete | Return cached response |
| Key exists, status=in_progress, <30s | Return 409 Conflict |
| Key exists, status=in_progress, >30s | Assume dead lock, retry |
| Key exists, status=failed, retryable | Allow new attempt |
| Key exists, request_hash mismatch | Return 422 Unprocessable |

* Retry with Exponential Backoff

** Retry Policy

#+begin_src clojure
(def retry-policy
  {:max-attempts 5
   :initial-delay-ms 100
   :max-delay-ms 30000
   :multiplier 2.0
   :jitter-factor 0.2
   :retryable-errors #{:network-timeout
                       :connection-refused
                       :rate-limited
                       :service-unavailable}
   :non-retryable-errors #{:card-declined
                           :insufficient-funds
                           :invalid-card
                           :fraud-detected}})
#+end_src

** Backoff Calculation

#+begin_src text
delay = min(max_delay, initial_delay * (multiplier ^ attempt))
jittered_delay = delay * (1 + random(-jitter_factor, +jitter_factor))
#+end_src

1. Preserve the original trace-id

** Failure Detection

| Condition | Counts as Failure |
|-----------|-------------------|
| HTTP 5xx response | Yes |
| Connection timeout | Yes |
| Read timeout | Yes |
| HTTP 4xx client error | No |
| Slow response (>threshold) | Partial (contributes to slow rate) |

** Fallback Behavior

When Stripe circuit is OPEN:
1. Log circuit state with trace context
2. If PayPal enabled and PayPal circuit is CLOSED, route to PayPal
3. If no fallback available, return 503 with Retry-After header
4. Emit ~circuit.open~ event to outbox (with traceparent)

** Webhook Processing

Stripe webhooks MUST:
1. Extract traceparent from ~metadata.traceparent~ if present
2. Create new trace if no traceparent (webhook is ingress point)
3. Validate webhook signature before processing
4. Publish events to outbox with trace context

#+begin_src clojure
;; Webhook handler
(defn handle-stripe-webhook [request]
  (let [event (verify-signature request)
        traceparent (get-in event [:data :object :metadata :traceparent])
        ctx (if traceparent
              (trace-spine/continue traceparent)
              (trace-spine/start-trace "webhook.stripe"))]
    (with-trace-context ctx
      (process-webhook-event event))))
#+end_src

* Wallet Credit Application Flow

** Pre-Authorization Check

Before applying wallet credit, verify:
1. Customer has sufficient balance
2. Wallet is not frozen/suspended
3. Order is eligible for wallet payment

** Application Flow

    Note right of Pay: applicable = min(order, available) = 75

    DB-->>Wallet: OK

    Wallet->>DB: INSERT INTO transactions (debit, order_id, ...)
    Wallet->>DB: COMMIT
    DB-->>-Wallet: OK
    Wallet-->>-Pay: {debit_id: wd_123, new_balance: 0}

    Note over Pay: Proceed with card charge for remaining $125

** Compensating Transaction

If the card charge fails after wallet debit:

#+begin_src clojure
(defn compensate-wallet-debit [ctx debit-id amount order-id]
  (with-trace-context ctx
    (let [span (create-child-span "wallet.compensate")]
      (try
        (wallet/credit {:debit-id debit-id
                        :amount amount
                        :reason :card-charge-failed
                        :order-id order-id})
        (finally
          (close-span span))))))
#+end_src

* Outbox Publisher

** Event Types

| Event Type | Trigger | Payload |
|------------|---------|---------|
| payment.initiated | Charge begins | order_id, amount, processor |
| payment.completed | Charge succeeds | order_id, charge_id, amount |
| payment.failed | Charge fails | order_id, error_code, reason |

** Transactional Guarantee

The outbox write MUST be in the same database transaction as the payment record update:

#+begin_src clojure
(defn complete-payment [ctx payment-id charge-result]
  (jdbc/with-transaction [tx db-spec]
    ;; Update payment record
    (jdbc/execute! tx
      ["UPDATE payments SET status = 'completed', charge_id = ?, updated_at = NOW()
        WHERE id = ?"
       (:charge-id charge-result) payment-id])

    ;; Write to outbox (same transaction)
    (jdbc/execute! tx
      ["INSERT INTO outbox (id, aggregate_id, event_type, payload, traceparent)
        VALUES (?, ?, ?, ?::jsonb, ?)"
       (random-uuid)
       payment-id
       "payment.completed"
       (json/write-str charge-result)
       (trace-spine/format-traceparent ctx)])))
#+end_src

* Error Handling

** Error Categories

| Category | HTTP Status | Retryable | Example |
|----------|-------------|-----------|---------|
| Transient | 503 | Yes | Processor timeout |
| Rate Limited | 429 | Yes (with backoff) | Too many requests |
| Client Error | 4xx | No | Invalid card |
| Fraud | 402 | No | Risk score exceeded |
| System | 500 | Depends | Internal error |
