* Overview

The Fraud Service provides real-time risk scoring for payment transactions. It
aggregates multiple signals (rules, velocity, blocklists, ML) into a single risk
score with sub-50ms p99 latency target.

All endpoints accept and propagate =traceparent= headers per L1-wire.org.

** Design Principles

1. *Low latency first* — Scoring happens in the critical payment path
2. *Fail open* — On service error, return neutral score (0.5) with error flag
3. *Trace correlation* — Every decision links to the originating trace-id
4. *Explainability* — Score response includes contributing signals

*** Decision Thresholds

| Score Range | Decision | Action                        |
|-------------+----------+-------------------------------|
| 0.0 - 0.3   | allow    | Proceed with transaction      |
| 0.3 - 0.7   | review   | Flag for manual review        |
| 0.7 - 1.0   | block    | Reject transaction            |

Thresholds are configurable per merchant via feature flags.

* Rule Engine Configuration

** Rule Schema

Rules are stored in PostgreSQL and cached in Redis.

#+begin_src sql
CREATE TABLE fraud_rules (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name          VARCHAR(255) NOT NULL UNIQUE,
    description   TEXT,
    condition     JSONB NOT NULL,
    score_delta   DECIMAL(4,3) NOT NULL CHECK (score_delta BETWEEN -1.0 AND 1.0),
    enabled       BOOLEAN DEFAULT true,
    priority      INTEGER DEFAULT 100,
    created_at    TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at    TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_fraud_rules_enabled ON fraud_rules(enabled) WHERE enabled = true;
#+end_src

** Condition DSL

Rules use a JSON-based condition language.

*** Operators

| Operator | Description                | Example                                           |
|----------+----------------------------+---------------------------------------------------|
| =eq=     | Equality                   | ={"eq": [{"field": "currency"}, "USD"]}=          |
| =gt=     | Greater than               | ={"gt": [{"field": "amount_cents"}, 50000]}=      |
| =lt=     | Less than                  | ={"lt": [{"field": "amount_cents"}, 100]}=        |
| =in=     | Set membership             | ={"in": [{"field": "country"}, ["RU", "NG"]]}=    |
| =match=  | Regex match                | ={"match": [{"field": "email"}, ".*@temp\\..*"]}= |
| =and=    | All conditions true        | ={"and": [...conditions]}=                        |
| =or=     | Any condition true         | ={"or": [...conditions]}=                         |
| =not=    | Negate condition           | ={"not": {...condition}}=                         |

* Velocity Checking

** Architecture

Velocity checks use Redis sorted sets for efficient counting with TTL-based
expiration.

#+begin_src mermaid :file velocity-architecture.png :eval never-export
flowchart LR
    subgraph Redis["Redis Cluster"]
        S1["velocity:user:{id}:hour"]
        S2["velocity:ip:{ip}:hour"]
        S3["velocity:device:{fp}:day"]
        S4["velocity:card:{bin}:hour"]
    end

    Scorer["Risk Scorer"] --> |ZADD + ZCOUNT| Redis
    Scorer --> |Pipeline| Redis
#+end_src

** Velocity Keys

| Key Pattern                      | Window | Limit | Score Delta |
|----------------------------------+--------+-------+-------------|
| =velocity:user:{id}:hour=        | 1h     |    10 |        0.10 |
| =velocity:ip:{ip}:hour=          | 1h     |     5 |        0.15 |
| =velocity:device:{fp}:day=       | 24h    |    20 |        0.10 |
| =velocity:card:{bin}:hour=       | 1h     |     3 |        0.20 |
| =velocity:email:{hash}:day=      | 24h    |     5 |        0.15 |

** Redis Operations

Each velocity check is a two-command pipeline:

#+begin_src clojure
;; Record event and count in single round-trip
(redis/pipeline
  (redis/zadd key timestamp transaction-id)
  (redis/zcount key (- now window-ms) now))
#+end_src

TTL is set to 2x window duration using =EXPIRE=.

* Blocklist Management

** Blocklist Types

| Type        | Storage      | Check Latency | Update Frequency |
|-------------+--------------+---------------+------------------|
| IP          | Redis Set    | <1ms          | Real-time        |
| Email       | Redis Set    | <1ms          | Real-time        |
| Email Hash  | Redis Set    | <1ms          | Real-time        |
| Device      | Redis Set    | <1ms          | Real-time        |
| BIN         | Redis Set    | <1ms          | Daily            |
| Country     | In-memory    | <0.1ms        | Deploy-time      |

** Redis Key Structure

#+begin_src text
blocklist:ip           -> SET of blocked IPs
blocklist:email        -> SET of blocked email hashes (SHA-256)
blocklist:device       -> SET of blocked device fingerprints
blocklist:bin          -> SET of blocked card BINs
#+end_src

* ML Signal Integration

** Feature Extraction

The fraud service extracts features for ML scoring but does not run the model
directly. Features are sent to a sidecar gRPC service.

** Feature Vector

#+begin_src clojure
{:user_account_age_days       (days-since (:created_at user))
 :user_previous_orders        (count (:orders user))
 :user_fraud_rate             (/ (:fraud_count user) (max 1 (:order_count user)))
 :transaction_amount_cents    amount
 :transaction_amount_zscore   (zscore amount user-avg user-stddev)
 :ip_country_match            (if (= ip-country billing-country) 1.0 0.0)
 :email_domain_age_days       (domain-age email)
 :device_seen_before          (if (known-device? device-fp user-id) 1.0 0.0)
 :session_duration_seconds    (/ session-duration-ms 1000)
 :hour_of_day                 (hour timestamp)
 :day_of_week                 (day-of-week timestamp)}
#+end_src

** ML Service Contract

#+begin_src protobuf
service FraudML {
  rpc Score(FeatureVector) returns (MLScore);
}

message FeatureVector {
  map<string, double> features = 1;
  string trace_id = 2;
}

message MLScore {
  double score = 1;
  string model_version = 2;
  map<string, double> feature_importance = 3;
}
#+end_src

** Fallback Behavior

If ML service is unavailable (circuit open), return neutral score (0.5) for the
ML signal. The rule engine and velocity checks continue independently.

** Span Attributes

Each span includes:

| Attribute                  | Type   | Description                     |
|----------------------------+--------+---------------------------------|
| =fraud.transaction_id=     | string | Transaction being scored        |
| =fraud.user_id=            | string | User making transaction         |
| =fraud.amount_cents=       | int    | Transaction amount              |
| =fraud.score=              | float  | Final or component score        |
| =fraud.decision=           | string | allow/review/block              |
| =fraud.rule.name=          | string | Rule name (if triggered)        |
| =fraud.velocity.key=       | string | Velocity check key              |
| =fraud.blocklist.type=     | string | Blocklist type (if matched)     |

** Error Handling

On any error, the span is annotated with:

#+begin_src clojure
{:error true
 :error.type "timeout"  ; or "redis_unavailable", "ml_circuit_open", etc.
 :fraud.fallback_score 0.5}
#+end_src
